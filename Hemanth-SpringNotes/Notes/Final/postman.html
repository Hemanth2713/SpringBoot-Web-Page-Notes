Postman Notes: API Testing Made Easy

A comprehensive guide to using Postman for API testing, inspired by Telusko tutorials

1. Introduction to Postman

Postman is a powerful tool for testing APIs by sending HTTP requests and analyzing responses.

It allows developers to interact with APIs without needing a frontend UI (like in React) and is widely used for testing server-side functionality.

[Image: Screenshot of Postman UI showing request panel. Generate Image: A screenshot of Postman with a GET request open.]

Why Use Postman?

Test if the server is sending/receiving the correct data.

Debug API issues quickly.

Supports various data formats (JSON, form-data, etc.).

No need to write frontend code to test APIs.

2. HTTP Methods in Postman

Postman supports various HTTP methods for interacting with APIs:

Method Description
GET Retrieve data from a server.
POST Send data to a server to create a resource.
PUT Update an existing resource on the server.
DELETE Remove a resource from the server.
HEAD Retrieve headers without the response body.
PATCH Partially update a resource.

Each method is selected from a dropdown menu in Postman’s request panel.

3. Sending a GET Request

A GET request retrieves data from a server using a URL (endpoint).

It’s the simplest way to test if an API is working.

Steps to Send a GET Request

1. Open Postman and create a new request.

2. Select GET from the HTTP method dropdown.

3. Enter the API endpoint (URL).

4. Click Send to execute the request.

5. Inspect the response (status code, JSON data, headers).

Example: Fetching User Data

Scenario: Fetch a list of users from a public API.

Endpoint: https://jsonplaceholder.typicode.com/users

Method: GET

Steps:

1. Set method to GET.

2. Paste the URL: https://jsonplaceholder.typicode.com/users.

3. Click Send.

Response:

- Status: 200 OK

- Body (JSON):

[
  {
    "id": 1,
    "name": "Leanne Graham",
    "email": "Sincere@april.biz"
  },
  ...
]

Use Case: Test if the server returns the correct user list without needing a browser or React frontend.

4. Sending a POST Request

A POST request sends data to the server to create a new resource.

Unlike GET, it requires a request body (e.g., JSON or form-data).

Steps to Send a POST Request

1. Select POST from the HTTP method dropdown.

2. Enter the API endpoint (URL).

3. Go to the Body tab.

4. Choose the data format (e.g., raw → JSON).

5. Enter the JSON data to send.

6. Click Send.

7. Check the response for success or errors.

Example: Creating a New User

Scenario: Create a new user on a mock API.

Endpoint: https://jsonplaceholder.typicode.com/users

Method: POST

Body (JSON):

{
  "name": "John Doe",
  "email": "john.doe@example.com",
  "phone": "123-456-7890"
}

Steps:

1. Set method to POST.

2. Enter URL: https://jsonplaceholder.typicode.com/users.

3. Go to Body → raw → Select JSON.

4. Paste the JSON data above.

5. Click Send.

Response:

- Status: 201 Created

- Body (JSON):

{
  "id": 11,
  "name": "John Doe",
  "email": "john.doe@example.com",
  "phone": "123-456-7890"
}

Use Case: Verify if the server accepts and processes the POST request correctly.

5. Using Postman vs. Browser

You can test GET requests in a browser by pasting the URL, but Postman offers more flexibility.

Browser

Pros:

- Quick for simple GET requests.

- No setup required.

Cons:

- Limited to GET requests (POST, PUT, etc., are not supported).

- No way to send custom headers or JSON body.

- Basic response display (hard to debug JSON).

Example: Paste https://jsonplaceholder.typicode.com/users in the browser to see the JSON data. But you can’t send a POST request.

Postman

Pros:

- Supports all HTTP methods (GET, POST, PUT, DELETE, etc.).

- Allows sending headers, query params, and body (JSON, form-data).

- Detailed response view (status, headers, body).

- Save requests for reuse.

Cons:

- Requires setup/installation.

Conclusion: Use a browser for quick GET tests, but Postman is essential for comprehensive API testing.

6. Postman vs. React

In a React app, APIs are called using fetch or axios, and the data is displayed in the UI.

Postman skips the UI and directly tests the API.

React

Pros:

- Visual UI to display API data (tables, cards, etc.).

- End-user experience testing.

Cons:

- Requires coding (fetch/axios, state management).

- UI bugs can interfere with API testing.

Example:

fetch('https://jsonplaceholder.typicode.com/users')
  .then(response => response.json())
  .then(data => console.log(data));

This fetches data and renders it in a React component.

Postman

Pros:

- No coding required.

- Focuses purely on API behavior (no UI distractions).

- Test raw JSON responses.

Cons:

- No UI visualization of data.

Example: Same GET request (https://jsonplaceholder.typicode.com/users) in Postman returns raw JSON without needing React.

Conclusion: Use React for frontend testing, Postman for backend API validation.

7. Testing JSON Data in Postman

Postman is ideal for testing if a server sends or receives the correct JSON data.

Example: Validating JSON Response

Scenario: Test if the server returns valid user data.

Endpoint: https://jsonplaceholder.typicode.com/users/1

Method: GET

Steps:

1. Send a GET request to the endpoint.

2. Check the response:

- Status: 200 OK

- JSON Body:

{
  "id": 1,
  "name": "Leanne Graham",
  "email": "Sincere@april.biz"
}

3. Verify if the expected fields (id, name, email) are present.

Example: Sending JSON Data

Scenario: Test if the server accepts a JSON payload.

Endpoint: https://jsonplaceholder.typicode.com/posts

Method: POST

Body:

{
  "title": "My Post",
  "body": "This is a test post.",
  "userId": 1
}

Response:

- Status: 201 Created

- JSON Body (includes new id): 

{
  "id": 101,
  "title": "My Post",
  "body": "This is a test post.",
  "userId": 1
}

Use Case: Ensure the server handles JSON correctly before integrating with a frontend.

8. Introduction to Swagger

Swagger (now part of OpenAPI) is another tool for API testing and documentation.

What is Swagger?

A tool to design, document, and test APIs.

Provides a UI (Swagger UI) to interact with APIs.

Often used by developers to share API specs with teams.

Swagger vs. Postman

Swagger: Focuses on API documentation and auto-generated UI based on API specs (e.g., OpenAPI YAML/JSON).

Postman: General-purpose API testing with more flexibility for ad-hoc requests.

Example: Swagger UI for https://jsonplaceholder.typicode.com would list endpoints like /users, /posts, and allow testing them in a browser-based interface.

When to Use Swagger?

To explore documented APIs.

To share API specs with teams.

For auto-generated testing UI.

Conclusion: Use Swagger for documented APIs, Postman for manual testing and flexibility.

9. Key Benefits of Postman

Ease of Use: Intuitive UI for sending requests.

Versatility: Supports all HTTP methods and data formats.

Testing: Validate server responses without a frontend.

Collaboration: Save and share requests with teams.

Automation: Write scripts to automate API tests.

10. Conclusion

Postman is an essential tool for developers to test and debug APIs.

It simplifies sending GET, POST, and other requests, inspecting JSON responses, and verifying server behavior.

While browsers can handle basic GET requests and React offers UI-based testing, Postman shines for backend-focused API validation.

Tools like Swagger complement Postman for documented APIs.
